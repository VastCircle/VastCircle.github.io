Meek:Re Superscar Processors

Apr 04, 2025
5497字

异构并行错误检测作为一种颇具前景的替代方案[22][23][2]，其核心思想是：利用强归纳法将运行在乱序执行超标量高性能核心（大核）上的软件程序划分为多个离散段（通过寄存器检查点/RCP实现），并在若干组能效比更高的小型核心（小核）上重新执行这些程序段以实现验证。为确保内存访问及其他不可重复操作的可重放性，系统会在大核的提交阶段从程序流中提取相关指令（如加载/存储操作）的地址与数据，生成分区式分布式加载存储操作日志。当某程序段的日志填满或达到指令超时阈值时，将触发新的寄存器检查点，对应的小核随即开始验证起始RCP（SRCP）与终止RCP（ERCP）之间的程序段。通过验证任务的流水线重叠执行，小核集群可提供充足的计算能力以匹配大核的执行速度，从而在低开销前提下实现全范围错误检测[22]。

**挑战。**传统锁步核通常采用完全相同的处理器核心搭配多路复用器和比较器实现[24]，而异构错误检测方案则依赖于大核与众多小核之间复杂的异步交互机制。该方案需要持续收集大核产生的加载/存储操作及周期性寄存器检查点数据，并将其动态分配给小核集群，同时灵活管理小核资源。这些数据必须从执行阶段持续保存至提交阶段，需确保按序传输并处理超标量提交操作。这也意味着系统必须以高带宽对采集数据进行优先级排序，并准确路由至对应小核，以防反馈压力导致大核执行停滞。现有架构研究[22][23]基于理想化微架构假设且未考虑操作系统支持，仅通过抽象仿真进行验证，这为实际部署设置了障碍。例如：如何在无需大幅改动微架构的前提下，从真实处理器核心的不同模块收集海量数据；如何在处理交叉通信竞争的同时实现跨芯片数据传输；以及如何为小核集群管理提供可配置框架——这些关键问题仍未得到解决。

主要贡献。我们通过硬件/软件协同设计重构技术方案，在系统各层级间实现功能权衡，证明仅需极少量修改即可在真实处理器核心上实现异构错误检测。该方案支持将大核上的应用线程通过寄存器检查点（RCP）分段后，在任意数量的小核上部署校验线程进行重放验证，同时仍允许其他线程正常执行（图1）。基于开源SoC平台[25]，我们构建了全栈式框架Make Each Error Count (MEEK)，实证表明：该架构可在成熟核心中以最小设计侵入度实现，且仅需对完整Linux系统进行数行代码修改。

MEEK

为实现高性能目标，我们必须在硬件与软件之间进行精心的设计划分（图2）。一方面，为实现小核对程序执行的重放，我们不得不采用硬件实现加载/存储日志记录功能——若采用基于软件的插桩技术[26]将导致难以承受的性能开销。另一方面，由于小核与主核异步执行页表遍历，操作系统必须以某种形式感知小核状态，因此我们让操作系统完全掌控小核调度：既避免了在非关键路径的硬件中实现复杂决策，又允许小核同时执行标准进程。

与完全透明的接口方案不同，为避免硬件层实现代价高昂的完整纠错机制[23]，程序通过与MEEK-ISA的协调函数封装进行交互：这些在main()函数前插入的协调函数负责向操作系统申请校验资源、验证检查输出，并在需要时调用错误处理代码。为此，我们对大核微架构进行了微调，在提交阶段插入只读观测通道（图2a），用于在每个RCP点采集大核状态数据（即架构寄存器文件、控制与状态寄存器），以及在RCP间捕获运行时数据（内存操作及其他不可重复操作的地址与数据）。

我们构建了专用数据互联架构（图2b），对提取的数据进行选择性广播/路由传输，最大限度降低数据通信带来的反压效应。在小核端，接收到的数据被缓冲至加载-存储日志（LSL）中——该结构在程序重放期间替代L1缓存，使小核能够：1）将架构状态重置至指定SRCP；2）精确重放RCP间的指令流；3）利用不同类型的数据（图2c）在ERCP点验证执行正确性。通过ISA接口（图2d及表I），可（重新）配置小核的检查特性，包括工作模式（应用模式/检查模式）等参数。

（DEU：数据提取单元；LSQ：加载-存储队列；PRFs：物理寄存器文件；CSRs：控制与状态寄存器；HM-NoC：半双工组播片上网络；MSU：模式切换单元；LSL：加载-存储日志；M-Bus：内存总线）

硬件方面： a) 在大核中部署了非侵入式DEU，在不干扰核心执行的情况下从不同位置收集数据； b) 开发了专用转发架构F2，优先处理数据并仅分发到相关小核； c) 设计了双模式小核，支持正确性检查或工作负载执行。

软件方面： d) 定制ISA为操作系统中的调度器e提供了抽象控制接口； e) 实现了对小核上验证和工作负载执行的灵活管理。

**错误检测方法。**通过使用校验线程实现并行化错误检测（图2(b)）：应用线程通过寄存器检查点（RCP）进行分段，分段触发条件包括目标加载-存储日志（LSL）填满、达到指令超时阈值或陷入内核模式。这些分段程序会由校验线程进行二次执行，其前提是假定之前所有分段都是正确的。重执行完成后，校验线程会将其架构寄存器状态与主应用线程在同一RCP处提供的寄存器状态进行比对。若寄存器匹配，则该分段被视为正确。若所有分段均通过校验，则判定整个执行过程完全正确。

III. THE MICROARCHITECTURE

尽管采用了第二节讨论的简化硬件功能集，MEEK仍需精心的微架构工程设计以实现高效、低开销的运行，特别是在数据传输方面。我们发现并改善了先前研究[22][23][27]因缺乏RTL实现而在分析和高层模拟中遗漏的多个关键瓶颈问题。同时，我们还识别出数据存储冗余导致的效率问题——转发路径所需的大部分信息在提交前本就缓存在核心内部，这意味着无需专用存储结构，直接从现有结构中转发数据即可。尽管存在这些瓶颈，我们证明只需对成熟异构SoC进行极小改动即可构建MEEK的微架构，避免了繁重的工程投入。我们将MEEK集成到开源异构SoC（Rocket Chip[25]）中，该系统同时包含高性能核心（BOOM）与高能效核心（Rocket）。通过Rocket Chip平台，我们验证了该方法对其他异构SoC的适用性，例如ARM的big.LITTLE[28]和Intel的P核与E核架构[29]。

A. 乱序执行超标量核心（大核）

程序重放需要从大核收集状态数据与运行时数据。这些数据在程序执行期间临时存储于核心内部，但分散在不同模块中：状态数据缓存于物理寄存器文件（PRFs）和控制状态寄存器（CSRs），而运行时数据则存储在加载-存储队列（LSQ）和CSRs中。虽然所有数据均可直接提取，但必须在狭窄时间窗口内完成——为确保指令顺序一致性，需在提交阶段（而非执行阶段）进行数据采集，且需在下一条指令覆盖数据前完成。

为此，我们开发了非侵入式数据提取单元（DEU）来建立观测通道（图3），该通道包含提交检测器（CD）、控制电路和旁路电路，被注入到PRFs、CSRs和LDQ中。CD通过监测重排序缓冲区（ROB）的指令提交，选择旁路电路来提取RCP点的数据或RCP间的运行时数据。该设计实现了及时数据提取，既无需额外缓冲器，也不改动现有寄存器路径。

以PRF为例的微架构实现

图3展示了以PRF（用于收集RCP数据）为例的微架构设计，其中DEU与ROB、PRFs及F2互联。当每条指令提交时：

  1. 操作码与功能码从ROB（图3a）路由至CD；
  2. CD据此判断是否需要提取数据（图3b）；
  3. 若到达RCP点，控制电路生成信号，抢占PRF控制器的权限；
  4. 将寄存器文件内容读取并转发至F2（图3c）。

PRF控制器在ROB与DEU间采用多路复用机制，当需要数据采集时DEU具有优先访问权，从而实现即时数据读取并避免数据覆盖。

C. 顺序执行标量核心（小核）

为实现线程级错误检测并确保校验线程与应用线程共存，小核微架构需支持多种操作模式，并提供软件可控接口。为此，我们升级了小核微架构（图4），新增模式切换单元（MSU，图4a）和加载-存储日志（LSL，图4b）。MSU作为控制引擎，LSL则缓存接收的数据包。在复制大核状态时，MSU会记录小核的架构寄存器文件，并用LSL中的状态数据替换它们。MSU还通过比对待执行线程与校验线程的线程ID（TID）来管理小核操作模式。当调度校验线程时，MSU切换操作模式，此时加载及不可重复指令的结果将从LSL获取。鉴于小核按顺序访问LSL，我们采用双路FIFO实现LSL存储体，较传统组相联架构降低了复杂度。

流水线集成

图4展示了LSL与MSU在5级流水线小核中的集成方案。LSL通过部署多路复用器接入内存访问（MA）阶段（图4b），并与加载-存储单元（LSU）的地址端口相连。该多路复用器根据MSU返回的操作模式选择性地将内存访问路由至LSL，并将TLB返回的虚拟索引与物理标签组合为LSL的地址端口。同时，MA阶段还集入了解复用器，将读取数据导回后续流水段。指令译码（ID）阶段部署了另一组复用/解复用器对，用于架构寄存器的记录与更新。最后，我们在MA阶段加入迷你译码器（Mini-D，图4c）以区分标准RISC-V指令和MEEK-ISA。

性能差距缓解

早期研究[22][23][27]因缺乏微架构细节，仅通过增加小核数量来弥补大小核性能差距。但在实际RTL实现中，Rocket小核的物理尺寸已接近大核的相当比例，原方案[22]采用12个小核的设计不可行（见V-E节）。我们发现，小核跟进大核执行速度的能力会因指令分布[35]产生显著差异。因此，最佳方案并非使用最小规格的Rocket核，而是针对BOOM核心的瓶颈进行平衡优化。具体措施包括：增加除法器展开度、扩展浮点单元流水线等关键组件规模。

四、指令集架构、操作系统与编程模型

我们详细说明了实现容错功能所需的ISA扩展。通过对内核进行少量代码修改，即可在维持其他线程标准调度与上下文切换的同时，为校验线程预留资源。

A. ISA支持

新指令分为大核（b.x()）与小核（l.x()）两类（表I）。b.hook()用于建立大小核关联，b.check()则通过开关DEU启用/禁用检测功能。小核指令l.mode()设置操作模式，l.record()/l.apply()对实现架构寄存器的记录与应用。为重定向PC到应用线程，我们改造标准跳转指令开发了l.jal()。通过将检查点终止视为类分支行为，流水线可自动处理PC变更引发的控制冲突。l.rslt()用于反馈RCP失配状态。鉴于b.hook()/b.check()可能引发小核资源争用，且l.mode()可能导致非法内存访问，这些指令被设计为需通过操作系统系统调用执行的特权指令。

B. 校验线程及其编程模型

校验线程通过构造函数和析构函数[36]与应用线程同步初始化（算法1第14行）。由于校验线程依赖LSL（加载-存储日志）中的FIFO结构重放内存操作，因此需要避免日志的上下文切换。在调度过程中，即使单个小核支持多线程调度，LSL资源也会被独占分配给指定的校验线程（算法1第12行）。LSL一旦被预留，系统将仅转发与该校验线程相关的数据，直至重执行完成。同理，绑定特定应用线程的校验线程在重执行完成前不可迁移。由于每个检查点大小有限（最多5000条指令），且执行完成后资源会立即归还操作系统重新分配，该机制不会造成资源饥饿问题。

编程模型 基于新指令集架构设计的校验线程极大简化了编码工作：

  1. 首先调用l.record()记录当前架构寄存器状态（算法2第15行），为后续验证提供基准；
  2. 创建忙等待循环监测LSL状态数据到达（算法2第19行）；
  3. 数据到达后通过l.apply()同步应用线程段状态（算法2第20行）；
  4. 使用l.jalr()重定向程序计数器至目标地址（算法2第21行）；
  5. 最后通过l.rslt()返回验证状态。若检测到错误，则触发中断通知操作系统（算法2第16-18行）。

C. 操作系统内核与验证机制

新指令集架构使得内核修改仅需集中在调度器的上下文切换函数，即可实现两大功能：

  • 检测能力配置
  • 校验线程管理[37][38]

根据大小核的差异化分工，系统对应用线程和校验线程采用不同的上下文切换策略：

  1. 应用线程进入切换时调用b.check(DISABLE)暂停检测（算法1第3行）；
  2. 退出切换时调用b.check(ENABLE)重新激活（算法1第20行）；
  3. 新线程调度时通过b.hook()建立与小核的绑定关系（算法1第10-13行）；
  4. 校验线程切换仅需使用l.mode()切换操作模式（算法1第3/7行），其余流程保持不变。

内核验证与死锁解决方案 MEEK的线程级验证机制可将OS内核视为特殊应用线程进行验证。但在开发过程中，我们发现早期研究[22][23][27]因缺乏操作系统评估而遗漏了一个死锁场景：由于SRAM日志容量有限，校验线程可能阻塞主线程直至小核完成执行，这种机制相当于小核持有大核需要的锁。若出现相反情况（大核持有小核需要的软件互斥锁），就会导致死锁。标准重执行过程中本不会出现这种情况，因为校验线程本身不访问内存，仅重放主线程获取锁时读取的内存内容。

然而在某些特殊场景下，校验线程可能意外需要实际内存读取：

  • 当小核发生指令错误（如超前大核执行，或指令在小核执行前被换出）[39]时
  • 缺页处理可能需要主线程持有的锁，从而引发死锁（图5a）

解决方案：

  1. 确保校验线程指令不会触发错误（如保持校验线程至少落后主线程1条指令），这样主线程会先遇到错误；
  2. 通过I/O同步机制，避免写出可能被未完成校验线程使用的页面（图5b）。实际采用方案一，通过DEU动态调节数据转发速率来维持指令间隔。
