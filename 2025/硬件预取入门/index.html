硬件预取入门

Apr 17, 2025
10137字

经典的预取策略需要在预取的激进型和准确性之间trade-off，评估包括两个指标：

  • Coverage (覆盖率)
    • 衡量预取成功占处理器请求的比例
  • Accuracy （准确率）
    • 衡量预取成功占所有预取请求的比例

放置预取的值

最古老最简单的软件策略是预取数据到core内的寄存器，当前主流的乱序处理器，只有在读数据指令产生依赖时才会stall，所以这种预取策略称为binding prefetch，因为在发起预取时，该值的用处就已经绑定。但是该方法也存在几个问题：

  • 会浪费宝贵的寄存器
  • 即使在内存负担很重的情况下也会强制进行预取
  • 在预取地址发生错误时很难处理，影响软件语义
  • 无法直接应用于指令预取

大多数硬件预取技术，则直接将预取的值放入cache或者额外的buffer以并行访问。在多核多处理器系统重，这些cache和buffer都需要参与到一致性协议中，所以在处理器真正使用该数据之前预取值所处的位置可能会更改，以保证每次访问的都是最新数据。这种预取策略也称为non-binding。此时，预取完全由硬件实现而不会影响软件语义。本文后续讨论的内容都基于硬件预取。

Next-line预取

Fetch-directed预取

branch-predictor-directed预取器可以复用现有的分支预测器来探测未来的控制流，FDIP(Fetcg-directed instruction prefetching)是目前做好的预取器之一，结构如下图。

FDIP将分支预测器从L1指令预取单元的stall中解耦，在两者之间添加了FTQ(Fetch Target Queue)。

  • prefetch使用FTQ中的地址来从L2 Cache中预取指令，并放置到较小的全相连buffer中。
    • 该buffer和L1 ICache并行的被fetch单元访问。
  • FDIP还会利用ICache空闲的读端口，判断FTQ中的指令是否已经在ICache中存在，只有miss的item才会被放入PIQ(Prefetch Instruction Queue)

解耦的BP和IF单元，FTQ（fetch target queue）是一个FIFO队列，缓存从BP发来的PB（predict block），然后发给IF单元。FTQ做为中间部件链接BP和IF单元。当L2总线空闲时，并且当前PB没有命中IC，通知PIQ（prefetching instruction queue），由PIQ向L2发起Cache prefetch请求。待到cache line返回后放入到PREB（prefetch buffer）。FTQ中首个PB同时查询IC和PREB首个entry，如果命中PREB则从PREB取数据，并将PREB数据写入IC。

优化 fully associative Prefetch Buffer with a sophisticated LRU replacement

现在将PREF设计为全相联，并且使用精确的LRU算法。这样一来FTQ中首个PB同时查询IC和所有PREB条目，如果命中PREB则从PREB取数据，并将PREB数据写入IC. 只有在预取缓冲区中有标记为可替换的条目，选择最近使用最少的可替换条目时，才会启动预取（在预取缓冲器中分配）。（一开始都会进行预取，后面会进行替换）

在分支预测错误时，所有条目都将被指定为可替换的，而不是像FIFO预取缓冲区那样刷新整个预取缓冲区时。当在指令高速缓存提取期间提取匹配的高速缓存块时，它们也将在预取缓冲区中标记为可替换的。

Filter Based on Number of FTQ Entries FTQ条目的距离首个条目越远，它就越有可能位于预测错误的路径上，因此在FTQ中更远的预取导致预取有用的概率降低并且潜在地浪费存储器带宽。目前发现，从FTQ前面的第二个条目开始，到FTQ中的10个条目提供了最佳性能。跳过FTQ中的第一个条目，因为当它接近从指令缓存中提取时，将其作为预取启动几乎没有好处。

2.3 不连续预取

最新案例是图2.6所示的不连续性预测器[21]，它通过维护一个"取指不连续性表"，将包含已采用分支的基本块PC映射到分支目标。类似实现据传已存在于某些商业处理器中。当下一行指令预取器在取指单元前探索时，会用每个块地址查询该表，匹配时既预取顺序路径也预取不连续路径。虽然该方案硬件简单且开销小，但存在两个主要限制：1）递归查询会导致预取块数指数增长，因此只能跨越单层不连续性；2）表项每个缓存块仅记录一个不连续性，而实际可能存在多个已采用分支的情况。

2.4 Prescient fetch

即通过利用空闲和并行资源来进行预取，一种方法是利用helper线程（helper线程在笔者上篇“多线程架构”中也有描述）来提前计算main线程的关键计算和控制指令区域所需的指令，帮助主线程更快的运行。

2.5 临时指令获取流

TIFS(Temporal Instruction Fetch Streaming)被设计用来解决herlper线程、fetch/discontinuity-cased机制的lookahead限制。其并不利用程序的控制流程图，而是通过记录L1 ICache过去的行为来直接预测未来的ICache miss。

如下图所示，L1 ICache的miss记录在指令miss log中，这是一个专用存储或者位于L2中。index table记录指令地址到上一次记录在log中位置的映射：

  1. L1 在地址C miss，后查询index table
  2. 映射到对应的miss log entry
  3. 获取stream 地址并访问对应的stream buffer
  4. 从L2中请求stream buffer中对应的block
  5. 返回对应内容
  6. 下一次发生访问地址D的miss时可以快速回应

特别的，其以cacheline粒度进行预测，而不像之前的方法以指令为粒度，很大程度上改善了lookahead的限制，并且可以跳过同一个cacheline内的控制流。

商业服务器工作负载跨越多个二进制文件、共享库和作系统代码，并表现出大型指令工作集，使 L1 指令缓存不堪重负

我们的主要观察结果是，重复的控制流图遍历会导致反复出现的指令缓存未命中序列，就像重复的数据结构遍历导致反复出现的数据缓存未命中序列一样。我们表明，几乎所有的指令缓存未命中，包括对非连续指令的控制转移，都是这种重复序列的一部分，我们称之为临时指令流。

  • 关键观察：L1指令不命中序列具有高度重复性（94%的不命中属于重复序列）。这些序列称为时间相关性指令流，跨越非连续分支（如函数调用、循环、条件分支）。
  • 核心机制：
    1. 记录不命中序列：在L2缓存中记录L1指令不命中的地址序列（称为Instruction Miss Log, IML）。
    2. 预测与预取：当检测到不命中的“头地址”时，从IML中读取之前记录的完整序列，提前预取后续指令块。
    3. 低开销实现：通过虚拟化技术将IML嵌入L2缓存，存储开销小于5%。
  • 硬件结构：
    • Streamed Value Buffers (SVB)：临时存储预取的指令块，避免污染L1缓存。
    • Instruction Miss Logs (IML)：记录L1指令不命中地址序列，存储在L2缓存中。
    • 索引表：嵌入L2标签数组，快速定位IML中的相关序列。
  • 操作流程：
    • 当发生L1指令不命中时，通过索引表找到最近一次相同地址对应的IML序列。
    • 按序列预取后续指令块至SVB，并根据实际访问情况动态调整预取速率。

个人感觉lml 是一张表 ， 一个条目保存一个地址 ， 然后通过index 去索引最近的地址，然后发出后续的访问

2.6 返回地址栈指向指令预取

RDIP(Return-addres stack-directed instruction prefetch)使用额外的程序上下文信息来增加预测的精度和lookahead。，基于两个观察：

  • call stack中的程序上下文和L1指令cache miss强相关
  • RAS(Return Address Stack)已经存在于所有高性能处理器中，其简单地总结了上下文。

RDIP将预取操作和RAS行程的签名相关联，存储在约64KB的signature table中，在每次调用和返回操作时都查看并触发预取。可以实现理想L1的90%加速。

核心问题

现代服务器应用中，L1指令缓存未命中率成为关键性能瓶颈（导致高达40%的性能下降）。传统顺序预取（如next-line）无法有效处理非连续性控制流（如函数调用/返回），而现有序列记录方法（如PIF）需要大存储开销（200kB+）和复杂纠错逻辑。

创新点

  1. 程序上下文表征
    • RAS签名生成：利用返回地址栈（RAS）状态生成紧凑签名（32位哈希），扩展方向位（call/return）区分不同调用上下文。
    • 签名更新时机：在call指令后（RAS push）和return指令前（RAS pop）更新签名，确保上下文精准关联。
  2. 缺失记录与压缩
    • Miss Table：组相联结构，记录签名与对应缺失地址序列。
    • 邻近块压缩：将连续缺失块压缩为基地址+位向量（如基地址+8位指示邻近块），减少存储开销（每条目16字节）。
  3. 上下文预测与预取
    • 前向签名预测：通过FIFO队列记录历史签名，关联当前签名与未来缺失序列，提升预取提前量。
    • 非关键路径操作：Miss Table访问和预取请求与指令流水线解耦，避免性能影响。

服务器与云计算应用：gem5模拟器、Hadoop（MapReduce）、Memcached（内存键值存储）、SSJ（Java服务）

RDIP包含三个主要组件：

  1. 提前返回地址栈(ERAS) - 跟踪将来可能的返回地址
  2. 过滤器 - 避免重复的预取请求
  3. 预取生成逻辑 - 根据返回地址生成预取请求

当处理器执行CALL指令时，系统会将返回地址推入ERAS。RDIP会定期检查ERAS中的地址，并为那些还未在缓存中的指令启动预取。

函数调用返回的指令缓存缺失问题

在示例中，论文展示了典型的函数调用和返回场景：

  1. 函数调用序列：
    • 主程序执行到某点调用函数 A
    • 函数 A 内部调用函数 B
    • 函数 B 内部调用函数 C
    • 函数调用链为：Main → A → B → C
  2. 返回地址栈(RAS)状态：
    • 当函数 C 执行时，RAS 中包含了三个返回地址：
      • 顶部：B+4（函数 B 中调用 C 后的下一条指令）
      • 中间：A+4（函数 A 中调用 B 后的下一条指令）
      • 底部：Main+4（主程序中调用 A 后的下一条指令）
  3. 缓存状态：
    • 由于程序局部性，C 的指令在缓存中
    • 但由于 Main、A、B 的代码可能已经被其他数据替换，返回地址所指向的指令很可能不在缓存中

论文通过这个示例说明了 RDIP 如何工作：

  1. 提前返回地址识别：
    • RDIP 会定期检查 RAS 内容
    • 发现返回地址 B+4、A+4 和 Main+4
  2. 预取操作：
    • 当函数 C 仍在执行时，RDIP 会提前预取 B+4 地址处的指令块
    • 随后提前预取 A+4 地址处的指令块
    • 最后提前预取 Main+4 地址处的指令块
  3. 好处：
    • 当函数 C 返回到 B 时，B+4 的指令已在缓存中
    • 当函数 B 返回到 A 时，A+4 的指令已在缓存中
    • 当函数 A 返回到 Main 时，Main+4 的指令已在缓存中
    • 整个返回过程中不会发生指令缓存缺失

数据的stride和stream预取

这类预取器捕获在虚拟地址中连续存放或者以固定stride存放的数据访问pattern，可以高效的处理矩阵、向量类数据结构的访问，但是对于指针类的数据结构性能较差。目前商业处理器中基本都具备这类预取器。

stride预取器实现的一个关键挑战是区分多个交错的stride序列，比如在矩阵-向量乘的场景中。下图展示了在加载指令时追踪stride的方案。其以PC作为tag，每个entry记录了加载的最后一个地址和前边两个entry间的地址差。每当观察到两个相同的stride时，就是用最后entry的地址和stride用于计算预取的地址。

# 《A Stateless, Content-Directed Data Prefetching Mechanism》论文分析

## 论文概述

这篇由Robert Cooksey、Scott Jourdan和Dirk Grunwald于2002年发表的论文提出了一种新的数据预取机制，称为"内容导向预取"（Content-Directed Prefetching，CDP）。该机制通过分析处理器加载的数据内容识别潜在的指针，然后预取这些指针指向的数据，从而减少内存访问延迟对程序执行的影响。

## 论文主要贡献

1. 提出了一种无需维护复杂状态的预取机制，可以直接从处理器加载的数据内容中识别指针
2. 开发了一种简单的实现方式，该方式无需对现有程序进行任何修改
3. 实验表明该方法对指针密集型应用程序的性能有显著提升

## 技术原理

CDP的核心思想是将处理器从缓存或内存加载的每个数据值都视为潜在的指针。工作流程如下：

1. 当处理器从缓存中加载数据时，CDP检查该数据是否可能是有效的内存指针
2. 如果数据看起来像指针，CDP会预取该指针指向的地址
3. 为避免过度预取，设计了过滤机制来减少不必要的预取请求

具体判断数据是否为指针的方法包括：
- 检查数据值是否落在有效虚拟地址范围内
- 检查该地址是否已映射到物理内存
- 应用各种启发式规则过滤无效预取

## 实现方式

论文描述了CDP的两种实现方法：

1. **硬件实现**：直接集成到处理器缓存控制器中
2. **软件实现**：通过系统模拟执行预取

硬件实现需要在L1缓存命中时附加一个小型预取引擎，检查数据内容并发起预取请求。整个预取过程对处理器透明，无需改变程序代码或编译器。

3.2.2 Pair-wise相关

correlation-based预取本质上就是一个lookup table，将当前访问的地址映射到之后可能连续访问的地址上去，所以其依赖于repetition，无法预取之前没有访问过的地址。

从本质上讲，基于相关的硬件预摘要是一个查找表，该表从一个地址映射到另一个地址，可能在访问序列中遵循它。尽管这种关联可以捕获顺序和大步的关系，但它更一般，例如捕获指针的地址与指向地址之间的关系。正是捕获指针遍历的能力，可以使相关的预赠与者比步步旅行前摘要更大的机会改善绩效的机会，因为在现代处理器上，指针拨入访问模式的速度不成比例。但是，寻址相关的预脱水依赖重复； 他们无法预拿以前从未被引用过的地址（与大步前摘要相比）。此外，地址相关预摘要需要巨大的状态，因为他们需要为每个地址存储继任者。因此，他们的存储要求与应用程序的工作集成比例地增长。地址相关的预定器设计的大部分创新都集中在管理这个巨大状态上。

意思是直接建立一个look-table 嘛 ， 本质可行，但是实际估计不太行 ，怎么可能映射所有的地址啊

3.2.3 Markov预取器

Markov预取器是最简单的利用pair-wise地址相关的设计。其直接实现了查找表，将trigger地址映射到其在off-chip访问序列中的下一个地址，如下图。

论文提出利用马尔可夫模型对内存缺失地址流（miss address stream）建模，通过历史访问序列预测未来可能访问的地址。具体来说：

  • 使用一阶马尔可夫模型（仅依赖前一次缺失地址）生成多步预测。
  • 通过硬件实现一个预测表，记录每个地址可能的后续地址（最多4个），并基于LRU（最近最少使用）策略对预测结果进行优先级排序。
  • 预取请求通过独立的预取缓冲区（Prefetch Buffer）交付给处理器，避免与常规缓存内容冲突。

预取器维护一个预测表，每个表项包含以下信息：

  • 当前地址（Key）：触发预取的地址（如最近一次缓存缺失的地址）。

  • 预测地址列表：最多存储 N 个后续地址（如4个），按访问频率或LRU优先级排序。

  • 状态信息：如访问计数器、时间戳（用于LRU替换）。

  • 马尔可夫模型说白了就是概率分布，根据整条链的出现概率来加权

  • 训练是实时的

  • 训练触发条件：每当发生 缓存缺失（cache miss） 时，当前的缺失地址会被记录为“父地址”，下一个缺失地址被记录为“子地址”。

  • 训练方式：

    • 若父地址不在预测表中，则创建新表项，并将子地址加入预测列表。

    • 若父地址已存在，则更新其预测列表（例如调整子地址的优先级或添加新子地址）。

  • 字段描述
    父地址（Key）触发预取的当前缓存缺失地址（如 0xABCD）。
    子地址列表最多存储 N 个预测地址（如4个），按优先级排序。
    访问计数器/时间戳记录每个子地址的访问频率或最近使用时间，用于优先级管理（如LRU）。
    有效位标记表项是否有效（避免无效地址占用资源）。

3.2.4 通过预取深度来提高lookahead

markov预取器的问题在于其有限的lookahead，即trigger地址和后续地址之间的时间太短，无法很好的隐藏访存延时。一种直接的方式是预取更多的地址来改善。

马尔可夫预取器的第一个关键限制是其前瞻性（lookahead）有限——即触发地址与后续地址因访问延迟导致的执行停顿之间的时间间隔过短，无法有效隐藏预取数据的访问延迟。

提升前瞻性的基本思路

一种直接的方法是预取更多地址，即沿着预测的全局地址序列进一步向前预取（类似于增加流预取器的深度）[41]。在类马尔可夫预取器架构中，可以通过递归查询预测表（基于初始预测地址生成更深层的预测）来实现更深的预取。然而，这种方法会导致以下问题：

  1. 高查询延迟：递归查询需要多次访问预测表，显著增加硬件延迟。
  2. 带宽压力激增：随着预取深度增加，对预测表的带宽需求呈几何级数增长。
  3. 候选地址爆炸：预测地址数量随深度指数增长，需引入策略限制其增长以维持准确性。

替代方案：流式存储与深度限制

为避免递归查询的开销，另一种方法是将预测表设计为折叠结构，在每个触发地址的表项中直接存储一个短序列的后续地址（称为“流”）[52, 53]。例如，每个表项可记录最邻近或最高概率的后续地址序列。但此方法需解决以下挑战：

  • 更新复杂性：一个地址可能需写入多个表项（例如需同时记录其直接前驱地址、前驱的前驱地址等）。
  • 固定深度限制：
    • 深度不足：若存储的序列长度过短（如仅存2-3个后续地址），可能截断长流，牺牲覆盖范围和前瞻性。
    • 深度过大：若序列过长，则浪费存储空间，且尾部地址可能因无关性降低准确性。
  • 实际场景多样性：研究表明，重复性流的长度差异巨大（从2次缺失到数千次缺失）[41, 42, 52, 54, 55]，需灵活适配不同长度模式。

优化策略：跳过无效预取阶段

流式预取的另一个问题是：流的起始部分地址可能无法及时预取。例如，触发地址后的前几个缺失地址距离触发时间过近，预取无法覆盖其延迟。此时，存储并预取这些地址会浪费资源并延迟对后续有效地址的预取。

解决方案：

  • 选择性跳过：忽略流的前几个地址，仅从能够提供延迟优势的首次缺失地址开始记录流。
  • 基于内存级并行性（MLP）的优化：
    • 在乱序执行核心中，指令窗口常导致多个片外缺失请求并发发出，执行因等待这些缺失返回而停顿。一旦缺失返回，指令窗口推进并计算依赖地址，允许下一组缺失并行发出。
    • Chou等人提出的时段关联预取（EBCP）[57]：
      • 将每组并行缺失（称为一个“时段”）中的首个缺失地址作为触发地址，用于查询下一时段（或后续时段）的缺失地址。
      • 核心思想：跳过那些在触发地址出现时可能已在传输中的缺失地址，优先预取更深层的地址。

3.2.5 通过死块预测来改善lookahead

3.3 空间相关预取

同样的，空间相关预取也需要有一个trigger event来发起预取，通常是一个memory access或者cache miss，其必须满足：

  • 提供key，以查找相关的layout pattern
  • 提供一个计算预取地址的基地址

3.4 基于执行的预取

其利用程序指令序列，在执行和退休前发现地址的计算和指针解析。其关键是要比指令执行本身运行的更快，要么是依赖于对地址计算的总结和猜测，要么是利用空闲的处理器资源在指令退休前探索。

3.4.1 算法总结

一些预取技术通过总结遍历数据结构的指令序列实现，以便比主线程执行更快以预取数据。

3.4.2 Helper线程和Helper core方法

3.4.3 run-ahead执行

Run-ahead execution利用那些造成长延时指令（如访问片外内存）期间暂停的core内执行资源。来提前探索其他的load miss等，实现预取。

3.4.4 Context restoration

随着虚拟化和多线程应用的普及，服务器端平凡的多线程或多OS切换，也会造成更多的cache miss，因为切换线程后，新线程的数据会破坏原有线程在cache中的存储。Context restoration则试图在线程切换时将cache中内容保存，切换回来后再加载复原。

3.4.5 计算传播

Computation spreading利用线程迁移技术，将跨越多个core的大的计算，分为几个类似的执行片段到每个core上，以提高cache访问的局部性。
